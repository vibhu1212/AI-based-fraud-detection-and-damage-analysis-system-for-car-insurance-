## 2024-04-03 - Eager Loading Conflict with Manual Fetches
**Learning:** In the `surveyor.py` API, there is a codebase-specific anti-pattern where SQLAlchemy relationships (`Claim.customer`, `Claim.icve_estimates`, `Claim.state_transitions`) are correctly eager-loaded using `joinedload` or `selectinload` on the base query, but then the API endpoint explicitly iterates over the results to manually extract IDs and fires off redundant bulk database queries (`db.query(...).filter(...in_(...))`) to construct in-memory lookup dictionaries. This explicitly negates the eager loading, effectively running the same queries twice and adding database/network overhead.
**Action:** When working with eager loading in SQLAlchemy, verify that the application code actually consumes the pre-fetched relationships directly through the object graph (e.g., `claim.customer.full_name`, `claim.icve_estimates[0]`) and aggressively delete any redundant manual bulk-fetching queries that recreate the same data in separate lookup dictionaries.
