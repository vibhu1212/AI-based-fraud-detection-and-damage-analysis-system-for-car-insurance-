## 2024-05-31 - Path Traversal Vulnerability in Storage Service
**Vulnerability:** The `StorageService` allowed users to control file paths via `object_key` or `filename` parameters without any server-side validation. This could allow an attacker to read, overwrite, or delete arbitrary files on the filesystem by passing absolute paths or relative paths with `..` directory traversal sequences. This was especially dangerous given the API endpoints that interact with file system paths directly, e.g., generating presigned URLs and storing PDFs.
**Learning:** Even when path generation is largely handled internally, operations like file upload, download, and deletion that accept keys must still strictly validate those paths to ensure they stay within the intended directory boundaries, especially in a local filesystem deployment scenario where S3 protections are not natively present.
**Prevention:** Every file operation MUST canonicalize and validate user-supplied paths. This is done by creating an `_is_safe_path` check that strictly rejects absolute paths (`/`, `\`, `:`), rejects `..` traversal sequences, uses `Path.resolve()` to resolve paths, and finally uses `Path.is_relative_to(base_path)` to ensure the resolved target file remains inside the permitted storage directory boundary.
